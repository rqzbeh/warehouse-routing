# پاسخ به سوالات فنی پروژه

## تحلیل فنی: منطق الگوریتم انتخابی

در این پروژه از مدل امتیازدهی وزن‌دار برای انتخاب انبار استفاده شده است. دلیل این انتخاب این است که ورودی مسئله شامل گراف واقعی مسیرها، یال‌های جاده‌ای، ظرفیت مسیرها یا داده لحظه‌ای مسیر بین nodeها نیست. بنابراین استفاده از الگوریتم‌هایی مثل Dijkstra یا A* در نسخه فعلی مبنای داده‌ای کافی ندارد.

الگوریتم فعلی ابتدا فقط انبارهایی را بررسی می‌کند که برای `sku` و `quantity` درخواستی موجودی کافی دارند. سپس برای هر انبار کاندید، هزینه نهایی با این عوامل محاسبه می‌شود:

- هزینه حمل اعلام‌شده از آن انبار تا مقصد
- فاصله جغرافیایی تقریبی انبار تا مشتری
- زمان تخمینی حمل
- ضریب ترافیک در بازه زمانی درخواست
- ضریب اولویت ناوگان توزیع سنگین
- جریمه عبور از زمان مورد انتظار مشتری

فرمول انتخاب به این شکل عمل می‌کند:

```text
TotalCost =
  adjusted_transport_cost
  + distance_penalty
  + eta_penalty
  + late_delivery_penalty
```

سپس انباری انتخاب می‌شود که کمترین `TotalCost` را دارد. مقدار `RouteOptimizationScore` هم از همین هزینه نهایی ساخته می‌شود تا خروجی قابل توضیح باشد.

در سناریوی نمونه، اگر فقط کمترین هزینه حمل خام انتخاب شود، انبار `karaj` انتخاب می‌شود؛ اما با درنظرگرفتن فاصله، ETA، ترافیک و اولویت ناوگان، هزینه نهایی آن از گزینه منتخب بیشتر می‌شود. نتیجه نمونه:

```text
tehran-west TotalCost = 114090.91
karaj TotalCost       = 144691.43
```

یعنی الگوریتم نسبت به انتخاب ساده بر اساس ارزان‌ترین هزینه حمل خام، حدود `21.15%` هزینه نهایی را کاهش می‌دهد.

Dijkstra زمانی مناسب‌تر است که داده مسیر به شکل گراف داشته باشیم؛ مثلا شهرها، جاده‌ها، زمان عبور، ترافیک هر یال و محدودیت خودروهای سنگین روی مسیرها. الگوریتم ژنتیک زمانی ارزش دارد که فضای تصمیم‌گیری بزرگ‌تر شود؛ مثلا چند سفارش هم‌زمان، ظرفیت ناوگان، چند مقصد، محدودیت شیفت راننده و زمان‌بندی چندمرحله‌ای. در مسئله فعلی، مدل اکتشافی وزن‌دار ساده‌تر، قابل توضیح‌تر و سریع‌تر است.

## چالش‌های طراحی مایکروسرویس و مدیریت Consistency موجودی

چالش اصلی در طراحی این سرویس این بود که انتخاب انبار و رزرو موجودی نباید فقط در حافظه برنامه انجام شود. اگر چند replica از سرویس در Kubernetes اجرا شوند، هر replica حافظه جداگانه دارد و نگهداری موجودی در حافظه می‌تواند باعث oversell شود.

برای حل این مشکل، سرویس stateless طراحی شده است. وضعیت موجودی فقط در PostgreSQL نگهداری می‌شود و برنامه در هر درخواست، کاندیدهای دارای موجودی را از دیتابیس می‌خواند.

رزرو موجودی با یک `UPDATE` شرطی انجام می‌شود:

```sql
UPDATE inventory
SET available_quantity = available_quantity - $3,
    reserved_quantity = reserved_quantity + $3,
    updated_at = now()
WHERE warehouse_id = $1
  AND sku = $2
  AND available_quantity >= $3
```

مزیت این روش این است که شرط `available_quantity >= quantity` داخل همان عملیات دیتابیس اجرا می‌شود. بنابراین اگر دو درخواست هم‌زمان برای آخرین موجودی یک کالا برسند، فقط یکی از آن‌ها موفق می‌شود و درخواست دیگر پاسخ `409 stock_changed` می‌گیرد.

چالش بعدی جستجوی سریع موجودی بود. برای همین روی جدول `inventory` ایندکس `sku, available_quantity` تعریف شده است تا سرویس بتواند سریع انبارهای دارای موجودی کافی را پیدا کند.

چالش دیگر این بود که API باید با ماتریس هزینه حمل کار کند. در نسخه نهایی، سرویس فقط وقتی تصمیم می‌گیرد که برای تمام انبارهای کاندید دارای موجودی، هزینه حمل ارسال شده باشد. اگر هزینه یک انبار دارای موجودی در request نباشد، درخواست نامعتبر در نظر گرفته می‌شود تا تصمیم ناقص یا اشتباه تولید نشود.

این طراحی باعث می‌شود سرویس برای scale-out آماده باشد: replicaهای مختلف state مشترک در حافظه ندارند، consistency موجودی در دیتابیس enforce می‌شود، و health/readiness endpointها برای اجرا در Docker و Kubernetes آماده هستند.
