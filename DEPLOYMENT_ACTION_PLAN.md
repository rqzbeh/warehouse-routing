# برنامه اقدام استقرار

## دامنه

هدف، استقرار یک میکروسرویس stateless برای تخصیص هوشمند انبار است. سرویس ورودی‌های موقعیت مشتری، SKU، تعداد، زمان مورد انتظار تحویل، هزینه‌های حمل‌ونقل و محدودیت‌های لجستیکی را می‌گیرد و خروجی شامل انبار منتخب، زمان تخمینی تحویل، هزینه نهایی و امتیاز بهینگی مسیر را برمی‌گرداند.

## تصمیم‌های اجرایی

- زبان: Go
- وب‌سرور: `net/http` از کتابخانه استاندارد
- دیتابیس: PostgreSQL
- اجرای محلی و تحویل پروژه: Docker Compose
- آمادگی Scale-out: سرویس stateless با probeهای `/healthz` و `/readyz`
- Kubernetes: مانفیست پایه برای Deployment، Service و HPA
- کش: فعلا حذف شده؛ فقط اگر p95 زیر ۲۰۰ میلی‌ثانیه با PostgreSQL ایندکس‌شده ممکن نبود، Redis اضافه می‌شود.

## فازهای انجام‌شده

### ۱. قرارداد API

- `POST /route`
- اعتبارسنجی JSON با رد فیلدهای ناشناخته
- محدودیت اندازه بدنه درخواست
- پشتیبانی از `expected_delivery_time` برای اعمال جریمه تاخیر نسبت به زمان مورد انتظار مشتری
- پاسخ خطای پایدار برای JSON نامعتبر، ورودی نامعتبر، نبود موجودی، تغییر موجودی و خطای دیتابیس

### ۲. مدل داده

- جدول `warehouses`
- جدول `inventory`
- ایندکس `inventory(sku, available_quantity)`
- ایندکس `warehouses(lat, lon)`

### ۳. الگوریتم تخصیص

- فیلتر انبارهای دارای موجودی کافی
- محاسبه هزینه بر اساس:
  - هزینه حمل
  - فاصله جغرافیایی
  - ضریب ترافیک
  - ضریب اولویت ناوگان
  - جریمه زمان تحویل
  - جریمه عبور از زمان مورد انتظار مشتری
- انتخاب کمترین هزینه
- خروجی explainable با نام‌های مطابق PDF: `WarehouseID`، `EstimatedDeliveryTime`، `TotalCost` و `RouteOptimizationScore`

`ponytail:` الگوریتم وزن‌دار قطعی سقف فعلی است؛ وقتی داده گراف مسیر واقعی اضافه شد، آن را به Dijkstra یا A* ارتقا بدهید.

### ۴. سازگاری موجودی

- رزرو موجودی با `UPDATE` شرطی انجام می‌شود.
- اگر موجودی بین انتخاب انبار و رزرو تغییر کند، سرویس `409 stock_changed` برمی‌گرداند.
- نیاز به lock در حافظه برنامه وجود ندارد، چون دیتابیس مرز سازگاری است.

### ۵. Containerization

- Dockerfile چندمرحله‌ای
- اجرای runtime با کاربر غیر root
- Docker healthcheck روی `/readyz`
- Compose شامل API و PostgreSQL
- پورت دیتابیس روی host publish نشده تا با PostgreSQL موجود روی VPS تداخل نکند.

### ۶. CI/CD

- GitHub Actions برای test، vet، SonarQube مشروط و build ایمیج
- Secretها داخل CI نگهداری می‌شوند، نه داخل repository

### ۷. Kubernetes

- `k8s/deployment.yaml`
- `k8s/service.yaml`
- `k8s/hpa.yaml`
- دیتابیس باید خارج از pod برنامه اجرا شود.
- migrationها نباید در هر startup کانتینر API اجرا شوند.

## معیار پذیرش

- `go test ./...` موفق باشد.
- `go vet ./...` بدون خطا باشد.
- Docker image روی VPS ساخته شود.
- `docker compose ps` سرویس‌های سالم را نشان دهد.
- `/readyz` پاسخ `204` بدهد.
- نمونه `POST /route` پاسخ معتبر JSON بدهد و موجودی را رزرو کند.
