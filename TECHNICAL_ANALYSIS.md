# تحلیل فنی و نکات استقرار

## هدف

سرویس برای انتخاب انبار مناسب بعد از ثبت سفارش نوشته شده است. ورودی شامل موقعیت مشتری، SKU، تعداد کالا، زمان درخواست، زمان مورد انتظار تحویل، هزینه حمل از انبارها و محدودیت‌های لجستیکی در بازه زمانی است. خروجی مطابق صورت مسئله با چهار فیلد `WarehouseID`، `EstimatedDeliveryTime`، `TotalCost` و `RouteOptimizationScore` برگردانده می‌شود.

## انتخاب‌های اصلی

- زبان پروژه Go است.
- لایه HTTP با `net/http` نوشته شده است.
- داده انبار و موجودی در PostgreSQL نگهداری می‌شود.
- سرویس stateless است و برای scale-out به state محلی وابسته نیست.
- اجرای پروژه با Docker Compose انجام می‌شود.
- برای Kubernetes سه مانفیست پایه `Deployment`، `Service` و `HorizontalPodAutoscaler` اضافه شده است.
- قرارداد API در `openapi.yaml` ثبت شده است.

## مدل داده

جدول `warehouses` شناسه، نام و مختصات انبار را نگه می‌دارد. seed اولیه شامل ۳۳ انبار نمونه است. جدول `inventory` موجودی آزاد و رزروشده هر SKU در هر انبار را ذخیره می‌کند.

ایندکس‌های اصلی:

- `inventory(sku, available_quantity)` برای پیدا کردن سریع انبارهای دارای موجودی
- `warehouses(lat, lon)` برای فیلتر یا مرتب‌سازی‌های مکانی بعدی

## الگوریتم انتخاب انبار

ابتدا انبارهایی انتخاب می‌شوند که برای SKU درخواستی موجودی کافی دارند. سپس برای هر انبار هزینه نهایی محاسبه می‌شود:

- هزینه حمل اعلام‌شده برای همان انبار
- فاصله تقریبی تا مشتری
- ضریب ترافیک
- ضریب اولویت ناوگان
- بازه زمانی معتبر برای محدودیت‌های لجستیکی
- جریمه زمان تحویل
- جریمه عبور از زمان مورد انتظار مشتری

انبار با کمترین هزینه انتخاب می‌شود. `RouteOptimizationScore` از روی همان هزینه نهایی ساخته می‌شود تا خروجی قابل توضیح باشد.

در این نسخه از امتیازدهی وزن‌دار استفاده شده است. دلیلش این است که ورودی چالش، گراف واقعی مسیرها را شامل نمی‌شود. اگر در نسخه بعدی داده مسیر به شکل node/edge اضافه شود، Dijkstra یا A* انتخاب مناسب‌تری است.

## سازگاری موجودی

رزرو موجودی با یک `UPDATE` شرطی در PostgreSQL انجام می‌شود. شرط `available_quantity >= quantity` داخل همان query قرار دارد؛ بنابراین دو درخواست هم‌زمان نمی‌توانند موجودی را منفی کنند. اگر موجودی بین انتخاب انبار و رزرو تغییر کرده باشد، سرویس پاسخ `409 stock_changed` برمی‌گرداند.

## تست و پذیرش

تست‌ها شامل این بخش‌ها هستند:

- تست الگوریتم انتخاب انبار
- تست اعتبارسنجی ورودی
- تست handlerهای HTTP
- تست یکپارچه PostgreSQL
- تست هم‌زمانی رزرو موجودی
- تست پذیرش مطابق ورودی و خروجی PDF
- تست بار k6 با آستانه p95 کمتر از ۲۰۰ میلی‌ثانیه

اجرای تست‌های اصلی:

```bash
go test ./...
go vet ./...
```

اجرای تست پذیرش:

```bash
python3 scripts/acceptance_test.py http://localhost:8080
```

اجرای تست بار:

```bash
BASE_URL=http://localhost:8080 SKU=PERF-SKU k6 run scripts/k6-route.js
```

## CI/CD

فایل `.github/workflows/ci.yml` روی push و pull request اجرا می‌شود. مراحل آن:

- اجرای تست‌ها
- اجرای `go vet`
- اجرای SonarQube در صورت تنظیم `SONAR_TOKEN`
- ساخت Docker image

## چالش‌های طراحی

تصویر PostGIS روی VPS مقصد برای معماری ARM64 در دسترس نبود. برای نیاز فعلی پروژه، PostgreSQL معمولی کافی است، چون الگوریتم از مختصات خام و فاصله تقریبی استفاده می‌کند. اگر جست‌وجوی مکانی سنگین‌تر شود، باید image سازگار با ARM64 برای PostGIS یا سرویس دیتابیس مدیریت‌شده انتخاب شود.

چالش دوم، سازگاری موجودی زیر بار هم‌زمان بود. این موضوع در لایه دیتابیس حل شده است، نه در حافظه برنامه، تا چند replica از سرویس هم رفتار یکسان داشته باشند.

چالش سوم، مدل‌کردن ضرایب ترافیک در بازه زمانی بود. برای همین `requested_at` در درخواست و `start_time`/`end_time` در محدودیت‌های لجستیکی پشتیبانی می‌شود. اگر بازه زمانی ارسال نشود، ضریب برای همه زمان‌ها معتبر در نظر گرفته می‌شود.
